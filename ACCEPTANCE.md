# loop-builder · 验收规格(ACCEPTANCE)

> 这是 loop-builder **自己的**验收目标,由 bagbag16 与 AI 用 loop-builder 的"验收目标共建"流程共建并确认(2026-06-22)。
> 用途:以后任何对 `SKILL.md` 的改动,都对照本规格判断"是否更好",而不是靠某一方主观感觉。
> 修订:本规格本身可被迭代,但只有"有理(新约束 / 新信息)且不违背设计理念"时才改,走 SKILL.md 的"目标变更纪律",并在此留痕。

## 被验收对象

loop-builder —— 一份"指导 agent / 人设计自治 loop"的方法论 skill。
面向:把会跑偏、会空转、会假性完成的长任务,装配成能独立验收、监管止损、能自停的闭环。

## 验收维度(必要性分级)

P 分级原则:**P0 = 破坏核心体验 且 高触发率**;非 P0 按 **阻塞 > 使用体验 > UI** 排。

| 维度 | 必要性 | 达标(可验证) | 防住的失败 |
|---|---|---|---|
| **稳定 / 鲁棒(可 loop)** | Must · P0 | 设计完整、能真正循环;每类阻塞都有处理方案;不空转、能自停 | 跑不起来 / 卡死 / 无限空转 |
| **专业性(导向更优 + 链路有力)** | Must · P0→P1 | P0:能导向比初始**更优**的结果(非原地打转);P1:链路清晰有力 → 更少轮次 / 消耗、最终效果达**专业**水准 | 能跑但产出平庸、低效 |
| **通用性** | Could · P2 | 步骤写清 + 门槛验证 → 跨领域可裁剪(预期自然达成,不单独重投入) | 只能用于个别场景 |
| **用户可观测简报** | Could · P2 | 产出用户可读简报,直观看中间变化、快速鉴别是否符预期(复用 ach 状态视图,不另造) | 用户看不见过程、无法鉴别 |

## 怎么验(对照 SKILL.md 的验收纪律)

- **独立验收**:换一个没参与改动的 agent,只拿 `SKILL.md` + 本规格,对抗式找"哪条维度不达标"(不是确认它好)。
- **能机械验就别靠主观**;终检含 **未见过的 holdout**(换个新领域任务做空跑)防针对可见验收集刷分。
- **Must 未达 = 不合格**(稳定/鲁棒、专业性的 P0 部分);Could 未达不阻塞交付,但要在结论里声明"已知牺牲"。

## 共建依据 / 出处

- 经 **3 轮对抗式自迭代**(独立验收 agent + 跨领域空跑实测:prompt 迭代 / 性能多目标 / 软目标文档)打磨,对抗 reviewer 终判"够好(9 条硬需求 7 PASS、0 FAIL)"。
- 维度与方法借鉴:ISO/IEC 25010、Kano、MoSCoW、Definition of Done;"自评不可靠"的处置借鉴 Reflexion / Voyager / DGM 的"独立验证 + holdout 防 Goodhart"。
